System Architecture: Interconnection Structure

The ecosystem is not a collection of independent tools — all components are connected via the LLM router proxy as central switchboard. Every request passes through routing, memory enrichment, source search with credibility scoring, and asynchronous quality evaluation, before and after a model responds.

Data flow of a typical request: Input → Stage 1: Rule-based dispatcher (domain recognition, token length, topic classification; no LLM call). Ambiguous cases → Stage 2: L1 reformulator (LLM classifies intent, recommends model, extracts search queries; overrides rule-based search-intent extraction). Routing → Neo4j slow memory (thematically relevant persons, concepts, projects injected as context). Routing → Session fast memory (themes already encountered). Model response → Synthegy evaluator (asynchronous quality score 1–5). High-stakes queries → adversarial second check (separate model).

LLM Router Proxy v3.3 — Core of the System

The two-stage routing is the central architectural decision. Stage 1 (rule-based dispatcher) classifies requests by domain keywords, token length and topic markers — without any LLM call, purely pattern-based. For simple queries (library searches, factual lookups, short code tasks) this stage routes directly. For ambiguous cases Stage 2 (L1 reformulator) is invoked: a smaller LLM classifies the query's intent, selects the appropriate large model, and extracts search queries for the retrieval layer.

The system has an internal slow memory (Neo4j knowledge graph with persons, concepts, projects, relations) and fast session memory (conversation-scope context from current session). Both are injected as system context before each model call. After each response, the Synthegy evaluator (a separate model instance) rates the response 1–5 on accuracy and groundedness. Scores below 3 trigger a retry or escalation.

Session Fast Memory

For each running conversation separately: topics already mentioned are compactly prepended. No database access — purely in the working memory of the running session.

Epistemic Tools

The anti-hallucination architecture follows He et al. (arXiv:2604.10511, 2026). Key components:

  • Source credibility scoring before retrieval (not only post-retrieval): sources are rated by recency, citation count, and document type
  • Synthegy evaluator pattern: a separate evaluator LLM rates each response on accuracy (1–5) and groundedness (1–5) before delivery to the user
  • Prior-forcing for counterintuitive questions: the system is prompted to explicitly consider the opposite of the obvious answer before committing (P4 structure: Prior Identification → Causal Mechanisms → Counter-evidence → Judgement)
  • Adversarial checking for high-stakes outputs: deepseek-r1:70b checks all formal mathematical claims independently via chain-of-thought

Search and Source Integration

The retrieval architecture integrates multiple search layers: LanceDB vector search (semantic + BM25 hybrid), Neo4j knowledge graph traversal, arXiv search (2.94M articles, MMR diversification), and PhilArchive search (187,882 philosophy preprints). The source credibility scoring system assigns scores based on document type (peer-reviewed > preprint > forum), recency, and citation count. All sources retrieved are logged with their credibility score for the Synthegy evaluator.

What Remains Experimental

  • LLM Router model-quality correlation: The qwen3:235b routing preference (prior-forcing method from He et al. 2026) is theoretically well motivated, but not empirically verified for this specific system.
  • Synthegy evaluator validation: The scores (1–5) are not calibrated against human judgements; it is unclear whether high scores correlate with actually useful responses.
  • Complete Lean 4 formalisation of sol_leftCancel: The only project-specific axiom in the Solid Modules project awaits the Mathlib lemma IsDiscrete → IsTiny.
Honest assessment: The system is functional and tested in daily research work. But it is a self-built single system without peer review, without A/B tests, and without systematic evaluation. The architectural decisions are well-motivated — the empirical confirmation is pending.

What Is Still Missing

  • A dedicated Plotinus store (Henry-Schwyzer edition or Loeb digitisation) for authentic primary text access beyond the TLG broad corpus
  • Systematic evaluation of the Synthegy evaluator against human judgements
  • The Mathlib lemma IsDiscrete → IsTiny for the complete sol_leftCancel proof

Synthegy-Evaluator (asynchrone Qualitätssicherung)

Nach jeder generierten Antwort bewertet ein dediziertes Evaluierungsmodell asynchron — ohne Latenz — ob die Antwort die gestellte Frage tatsächlich beantwortet und ob Widersprüche zum injizierten Kontext vorliegen. Vergibt einen Score von 1 bis 5. Scores werden persistent im Neo4j-Wissensgraphen gespeichert, verknüpft mit den thematischen Domänen der Session — damit entsteht über Zeit ein Qualitätsprofil pro Domäne.

Lokale SearXNG-Instanz aggregiert parallel mehrere Engines: Brave Search · DuckDuckGo · Wikipedia (mehrsprachig) · Semantic Scholar · arXiv · Google Scholar · PubMed. Alle Suchergebnisse fließen mit Credibility-Annotation in den Kontext ein. Mojeek, Yahoo, Google und Bing sind deaktiviert (Rate-Limiting).

Parallel laufen direkte Abfragen an: OpenAlex · CORE (Open Access) · Open Library · lokalen arXiv-Index.

Perplexica (KI-gestützte Websuche)

Eigenständige Perplexica-Instanz als Docker-Container — eine Perplexity-artige Suchoberfläche, die natürlichsprachige Anfragen mit SearXNG-Websuche kombiniert und Quellenangaben direkt inline liefert. Läuft separat vom Proxy-System, für direkte interaktive Webrecherche nutzbar. Docker Desktop ist installiert; Perplexica ist als Dienst konfiguriert.

Hybrider arXiv-Index (2,4 Millionen Einträge)
Ein lokaler Index mit ca. 2,4 Millionen arXiv-Einträgen kombiniert BM25-Schlüsselwortsuche mit semantischer Vektorsuche. Maximal Marginal Relevance (MMR) erhöht die thematische Diversität der Suchergebnisse. Fuzzy-Autorensuche behandelt Namensvarianten korrekt. Der Index deckt Mathematik, Physik, Informatik und angrenzende Disziplinen ab.

Experimentelle Mathematik

Ein dediziertes WebUI-Werkzeug implementiert den Borwein/Bailey-Workflow für experimentelle Mathematik. Philosophie: mathematische Entdeckung über numerische Exploration, Mustererkennung und rigorose Nachverifikation — die Methode, mit der BBP 1996 via PSLQ entdeckt wurde.

PSLQ und Konstantenerkennung

  • Hochpräzisionsarithmetik bis 500+ Dezimalstellen via mpmath
  • PSLQ-Konstantenerkennung: findet ganzzahlige Relationen (z.B. ζ(2) = π²/6)
  • Basis: π, e, log, sqrt, ζ, Γ, euler, catalan, φ und Kombinationen
  • Rigorose Intervall-Arithmetik mit garantierten Fehlerschranken

BBP-Formel und Π-Berechnung

  • BBP-Formel: beliebige Hexadezimalstellen von π direkt (ohne alle vorherigen)
  • Vollständiger Borwein/Bailey-Zyklus: Hochpräzision → identify → PSLQ → LLM-Interpretation
  • Systematische Gegenbeispielsuche (LLM generiert Testcode)

PARI/GP-Integration (Version 2.17.3)

Direktzugriff auf PARI/GP für zahlentheoretische Berechnungen: factor · bnfinit (Klassengruppen) · ellinit / ellrank (elliptische Kurven) · lfun (L-Funktionen) · polroots · Modulformen. Cohen GTM 138-Algorithmen verfügbar. Direktes WebUI-Tool pari_rechnung.

Dokumentenverarbeitung und Schrifterkennung

Chandra OCR 2 (v0.2.0)

5-Milliarden-Parameter-Modell für Dokumentverständnis. Strukturierter Output: Markdown · HTML · JSON mit erhaltenem Layout. 90+ Sprachen inkl. Handschrift, Tabellen, mathematische Formeln. Inference-Modi: lokal via HuggingFace (--method hf) oder via vLLM-Server. Systemweiter Wrapper ~/bin/chandra_ocr. Vollständig ohne Cloud-Beteiligung.

LLM-gestütztes Dokumentenverständnis

Ergänzend zu Chandra OCR: Beliebige Ollama-Modelle für Dokumentinterpretation, Layout-Analyse, inhaltliche Erschließung. Gemma4:26b verarbeitet nativ Bilder und PDFs (256K Kontext, Vision). Die Kombination OCR-Extraktion + LLM-Interpretation ermöglicht vollständige Erschließung von historischen Manuskripten bis wissenschaftlichen Artikeln — ohne Cloud.

Was noch fehlt

Für eine belastbare Beurteilung des Systems fehlt bisher eine systematische quantitative Evaluierung — dazu war schlicht keine Zeit. Offen sind:

  • Retrieval Recall und Precision über alle Vektordatenbanken — kein systematischer Benchmark, nur qualitative Einschätzung
  • Latenz-Profiling je Modell und Anfrage-Typ — keine gemessenen End-to-End-Latenzen unter realen Bedingungen
  • MMR-Impact-Messung — erhöht Maximal Marginal Relevance die thematische Diversität der Suchergebnisse tatsächlich messbar? Nicht evaluiert.
  • Speicher- und Throughput-Benchmarks — keine quantifizierten RAM-Footprints je Modell unter gleichzeitiger Last
  • Vergleich gegen naive Baselines — kein Vergleich gegen einfache Keyword-Suche oder direkten LLM-Aufruf ohne Kontext-Injektion. Der Nutzen des zweistufigen Routings ist empirisch nicht bestätigt.
  • Fehlerprofile nach Domäne — keine systematische Fehleranalyse: wo halluziniert das System, wo gibt es Lücken, welche Domänen sind schwächer abgedeckt?
  • Empirische Halluzinationsraten vor und nach dem Anti-Halluzinations-Block — der Block wurde auf Basis qualitativer Einschätzung eingeführt, nicht auf Basis gemessener Ratenreduktion
  • False-Positive-Rate im Safety-Classifier (llama-guard3:8b) — keine Messung, wie oft legitime Anfragen fälschlich blockiert werden
  • Messbarer Nutzen des Ensemble-Checks gegenüber Single-Model-Antworten — das Cross-Modell-Disagreement-Signal (He et al. 2026) ist theoretisch gut begründet, aber für dieses spezifische System nicht empirisch verifiziert
  • Quantifizierter Effekt der Prior-Forcing-Methodik in den epistemischen Werkzeugen — verbessert die P4-Struktur tatsächlich die Antwortqualität bei gegenintuitiven Fragen? Nicht gemessen.
  • Synthegy-Evaluator-Validierung — die Scores (1–5) sind nicht gegen menschliche Urteile kalibriert; es ist unklar, ob hohe Scores mit tatsächlich nützlichen Antworten korrelieren
  • Vollständige Lean 4-Formalisierung von sol_leftCancel — das einzige projektspezifische Axiom im Solid-Modules-Projekt wartet auf das Mathlib-Lemma IsDiscrete → IsTiny
Ehrliche Einschätzung: Das System ist funktional und in der täglichen Forschungsarbeit erprobt. Aber es ist ein selbstgebautes Einzelsystem ohne peer review, ohne A/B-Tests und ohne systematische Evaluierung. Die Architekturentscheidungen sind gut begründet — die empirische Bestätigung steht aus.
